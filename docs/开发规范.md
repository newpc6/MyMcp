# 开发规范

## 后端分层

后端新增功能按 `API -> Service -> Repository -> Model` 分层实现。

- API 层只负责路由、参数接收、权限依赖、响应封装和审计日志触发。
- Service 层负责业务流程、业务校验、文件处理、外部进程/外部接口调用等编排逻辑。
- Repository 层负责数据库查询、新增、更新、删除、分页和批量更新。
- Model 层只定义 ORM 结构和关系。

## 数据库访问约束

- 新增功能不要在 API 层直接操作数据库。
- 新增功能的 Service 层尽量不要直接写 `select`、`db.get`、`db.query`、`db.add`、`db.delete` 等数据库操作，应封装到 `app/repositories`。
- 历史代码可逐步迁移；新增代码优先遵守本规范。
- Repository 方法只做数据访问，不包含文件系统、HTTP 调用、进程管理等业务副作用。

## Seed 初始化

- Seed 入口统一为 `app/services/seed_service/__init__.py` 中的 `ensure_seed_data`，外部只调用该入口。
- 基础框架 seed 放在 `app/services/seed_service/base.py`，包含工作台、系统管理、基础配置、默认角色和默认管理员。
- 每个业务模块单独维护自己的 seed 文件，例如算法中心放在 `app/services/seed_service/algorithms.py`。
- 菜单 upsert、配置 upsert、默认角色菜单同步等通用逻辑放在 `app/services/seed_service/common.py`。
- 新增业务模块不要继续堆到基础 seed；新增菜单、配置、初始化表结构时，在自己的模块 seed 中实现，并在入口中按依赖顺序调用。
- Seed 必须保持幂等，不覆盖用户已修改的配置值，不覆盖非历史菜单图标、颜色和样式。

## 文件和大资源

- 算法包、模型文件、日志、构建产物等大资源不要提交到 Git。
- 上传型算法包保存在 `server/algorithm_plugins`。
- 上传型模型包保存在 `server/algorithm_models`。
- Watchdog 需要忽略算法包、模型文件、日志、构建产物等目录，避免上传资源导致后端重启。

## 算法插件

- 平台侧对外只暴露统一运行时接口，例如 `/api/algorithm-runtime/{category}/{capability}/invoke`。
- 插件 SDK 自身端口只作为平台后端内部转发地址，不直接暴露给前端页面或业务调用方。
- 算法代码包和模型版本分开管理。算法包负责接口和逻辑，模型版本负责模型文件路径、来源、版本、状态和绑定关系。

## 前端分层

前端新增功能按 `页面(view) -> 组件(component) -> API 封装(api) -> Store` 分层实现。

- `views/` 下放路由级页面，每个业务模块独立子目录，路由级页面只负责数据编排和页面骨架。
- `components/` 下放可复用组件，业务相关组件放在对应业务子目录（如 `components/mcp-auth/`），通用组件直接放在 `components/` 根目录。
- `api/` 下每个后端模块对应一个文件，统一返回 `Promise<ApiResponse<T>>`，禁止在组件里直接 `axios.get`。
- `store/` 只放跨页面共享的状态，页面私有状态用 `ref/reactive`，不要把所有东西都塞进 Pinia。

## 目录结构约定

```
frontend/src/
├── api/              # 接口封装，按后端模块拆分
├── assets/          # 静态资源（图片、字体）
├── components/       # 可复用组件
│   └── {业务}/       # 业务组件按模块归类
├── composables/      # 组合式函数（useXxx）
├── layout/           # 布局骨架（MainLayout 等）
├── router/           # 路由配置
├── store/            # Pinia store
├── styles/           # 全局样式与设计 token
│   ├── variables.scss    # 设计 token（颜色、间距、字号）
│   └── components/       # 组件级公共样式
├── types/            # TypeScript 类型定义，按模块拆分
├── utils/            # 工具函数（request、format、copy 等）
└── views/            # 路由级页面
    └── {业务}/       # 业务页面按模块归类
        └── components/   # 仅当前页面用的子组件
```

- 跨页面复用的子组件提到 `components/`，仅当前页面用的子组件放在 `views/{业务}/components/`，不要全部堆在 `components/` 根目录。
- 工具函数优先抽成 `composables/useXxx.ts`，避免在多个页面里复制粘贴逻辑。

## 样式与设计系统

- 设计 token 统一在 `styles/variables.scss` 维护，颜色、间距、字号、圆角、阴影、z-index 都走 token，禁止在业务代码里写魔法数字。
- 颜色统一用 CSS 变量（`var(--color-primary)` 等），支持亮/暗主题切换；禁止在组件里硬编码 `#1890ff` 这类色值。
- 主色、语义色（success/warning/danger/info）只允许从 token 引用，不要自定义新的语义色。
- 间距统一用 4 的倍数（4/8/12/16/24/32），优先使用 Tailwind 工具类（`p-4`、`gap-2`），自定义 SCSS 只在工具类无法表达时使用。
- 圆角统一：卡片 `12px`，按钮 `8px`，输入框 `8px`，弹窗 `16px`，不要出现 `2px`、`4px`、`20px` 混用。
- 阴影统一用 token（`--shadow-sm/md/lg`），禁止业务代码里随手写 `box-shadow: 0 4px 12px rgba(...)`。
- 禁止滥用渐变：默认状态用纯色，仅在强调态（如主按钮、品牌区）使用 token 内定义的渐变，不要每个卡片头部都加渐变。
- 动效克制：hover 只做颜色/背景过渡（`transition-colors`），不要叠加 `transform: translateY(-2px)` + `box-shadow` + `rotate` 这种堆叠效果。

## 布局规范

- 全局统一使用左侧边栏 + 顶栏布局（`layout/MainLayout.vue`），顶栏只放面包屑、用户菜单、主题切换、全局搜索。
- 路由级页面统一包在 `<PageContainer>` 里，统一 padding 和最大宽度，不要每个页面自己写 `padding: 20px`。
- 页面标题与操作区使用统一的 `PageHeader` 组件，标题在左、操作按钮在右，禁止每个页面自己拼 flex 布局。
- 表格、表单、卡片间距统一走 token，不要出现 `mt-5`、`mt-10`、`mt-15`、`mt-16`、`mt-20` 混用，统一用 Tailwind 间距。

## 组件规范

- 表单：使用 `el-form` + `el-form-item`，校验规则抽到 `rules` 常量，不要写在模板里。
- 表格：分页、查询、loading 状态统一用 `useTable` composable 封装，不要每个页面重复写 `currentPage/pageSize/total/handleSizeChange`。
- 弹窗：标题、宽度、footer 按钮顺序统一（取消在左、主操作在右），宽度优先用预设档位（`sm/md/lg`）。
- 按钮：主操作 `type="primary"`，次要操作默认，危险操作 `type="danger"`，链接型用 `link`，不要混用。
- 图标统一从 `@element-plus/icons-vue` 引入，不要混用多套图标库。

## TypeScript 规范

- 所有 `.vue`、`.ts` 文件使用 `<script setup lang="ts">`，禁止用 Options API。
- 接口返回类型在 `types/` 下定义，禁止用 `any` 兜底；确实无法定义类型的用 `unknown` 再收窄。
- API 函数必须声明入参和返回类型，禁止 `api.get('/xxx')` 不带泛型。
- 公共类型用 `interface`，联合类型/字面量类型用 `type`，不要混用。

## 主题与暗色模式

- 主题切换通过 `<html class="dark">` 切换，所有颜色走 CSS 变量，禁止在业务代码里写 `:root` 覆盖。
- 新增组件必须同时验证亮/暗两种主题，不要只测亮色。
- Element Plus 主题通过 `styles/element-theme.scss` 覆盖，不要在业务组件里 `:deep(.el-xxx)` 改颜色。

## 性能与质量

- 路由组件全部用 `defineAsyncComponent` 或 `() => import()` 懒加载，不要在 `router/index.ts` 里同步 import 业务页面。
- 大列表用虚拟滚动（`el-virtual-list` 或 `el-table-v2`），不要无脑 `v-for` 渲染上千条。
- 图片资源走压缩，超过 200KB 的图片必须压缩或走 CDN，不要直接提交原图。
- 提交前必须 `npm run lint` 通过，禁止提交有 warning 的代码。
